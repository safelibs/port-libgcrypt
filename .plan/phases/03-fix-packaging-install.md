# Implementation Details

## Workflow-Wide Requirements For This Phase

- Consume the listed preexisting inputs in place. Existing source snapshots, metadata, tests, packages, validator checkouts, and prior artifacts are inputs, not rediscovery or regeneration tasks unless this phase explicitly rebuilds generated package/validator artifacts.
- Keep `validator/`, `validator-local/`, and `validator-artifacts/` external to the port repository. Do not commit nested validator contents or generated validator artifacts.
- Preserve the Rust toolchain contract: root `rust-toolchain.toml` and `safe/rust-toolchain.toml` must stay identical and pinned to Rust `1.85.1` with `profile = "minimal"` and `components = ["rustfmt"]`.
- Run `safe/scripts/check-rust-toolchain.sh` before every Cargo, release-library, Debian package, or dependent safe-image build.
- Build any package artifacts used as validator inputs from a clean phase commit after `phase/impl_p03_fix_packaging_install` points at `HEAD`. If tracked files or `validator-report.md` change after validation, amend the same phase commit, force-update the phase tag, rebuild packages, regenerate `validator-local`, and rerun the phase validator commands until the worktree is clean and artifacts match the final phase commit.
- `validator-report.md` must identify the safe port by implement phase ID and `phase/impl_p03_fix_packaging_install`, not by embedding the literal hash of the final report commit.

## Phase-Specific Details

- Consume the phase 2 failure inventory. Do not rerun full discovery in this
  phase; rerun only the focused package-gating validator checks listed above.
- Fix all `packaging-install` failures before source API, symmetric/digest, or
  public-key phases run. Later phases must be able to assume that package
  installation, development metadata, and override evidence are already clean.
- Treat these as `packaging-install` failures: missing
  `override_debs_installed`, missing or mismatched
  `override_installed_packages`, package install errors, absent runtime or dev
  package files, wrong package name/version/architecture/source metadata,
  missing `gcrypt.h`, missing or broken `libgcrypt.so` or
  `libgcrypt.so.20`, missing symbols, broken `libgcrypt-config`, broken
  `pkg-config libgcrypt`, Multi-Arch metadata problems, dependency problems,
  and package scripts that remove or replace the safe library incorrectly.
- Add `safe/scripts/check-validator-package-dev-probe.sh` if it does not already
  exist. The script must accept `--dist`, `--override-root`, `--port-lock`,
  `--artifact-root`, and optional `--image` defaulting to `ubuntu:24.04`; fail
  unless `--override-root/libgcrypt` contains exactly the two `.deb` files named
  in the lock; validate their SHA256 and size before Docker starts; install the
  local `.deb` files in a clean container with the minimal compiler and
  development dependencies; reject any installed `libgcrypt20` or
  `libgcrypt20-dev` package whose package name, version, architecture, filename,
  size, or SHA256 does not match the lock and mounted files; compile and run a
  reduced package-surface C probe that includes `gcrypt.h` and calls only
  `gcry_check_version(NULL)`; compile and run the same package-surface probe
  once with direct linker inputs, once with
  `pkg-config --cflags --libs libgcrypt`, and once with
  `libgcrypt-config --cflags --libs`; record the container log, compiler
  commands, tool outputs, installed package metadata, and probe result under the
  requested artifact root; and exit nonzero on every package install, metadata,
  compile, link, or `gcry_check_version` runtime failure. The phase 3 probe must
  not call `gcry_md_hash_buffer`, `gcry_create_nonce`, cipher, MPI, public-key,
  or other source-behavior APIs as gating checks; those compatibility failures
  remain in the phase 4 or later validator buckets.
- Do not fix source API behavior in this phase. If a source testcase fails after
  successfully installing the safe override packages and compiling/linking its
  probe, leave that testcase in the `source-api` bucket for phase 4.
- Add a minimal regression before each safe-side package/install fix.
  Regression candidates include installing the locally built `.deb` files in a
  clean package context, checking `dpkg-query` package metadata, compiling and
  linking a reduced C program with `gcrypt.h`,
  `pkg-config --cflags --libs libgcrypt`,
  and `libgcrypt-config --cflags --libs`, and checking `gcry_check_version`
  through the installed shared library.
- If phase 2 recorded zero `packaging-install` failures, do not invent a
  regression or package change. Rerun the package/development probe and focused
  phase 3 validator checks with override evidence, update
  `validator-report.md` with the clean bucket result, package/development probe
  result, and artifact roots, and make the required
  `impl_p03_fix_packaging_install:` phase commit. If the report was already
  exact and no tracked file changed, create the phase commit with
  `git commit --allow-empty` before tagging it.
- If the validator itself has a packaging/install harness bug, document the
  exact defect in `validator-report.md`. Do not edit validator files. Use the
  testcase skip file only for an exact testcase execution bug; proof or
  inventory bugs are report-only skips.
- Commit fixes and report updates. End with one commit whose subject begins
  `impl_p03_fix_packaging_install:` and tag it
  `phase/impl_p03_fix_packaging_install`.

# Success Criteria

- Package install, header, linker, `libgcrypt-config`, pkg-config, Multi-Arch,
  dependency, symbol/SONAME, and override-evidence failures from phase 2 are
  fixed or documented as exact validator bugs.
- The clean-container package/development probe passes with local safe
  `libgcrypt20` and `libgcrypt20-dev` installed from the validator override root,
  and the probe artifacts show successful direct-header, pkg-config, and
  `libgcrypt-config` compile/link/run paths.
- Focused source and usage smoke validator runs produce port-mode artifacts with
  verified safe override package installation evidence for every non-skipped
  testcase. Source or usage compatibility failures with complete install
  evidence remain assigned to phase 4 or later buckets rather than blocking
  this packaging phase.
- Existing safe build, ABI, package metadata, dependent metadata, and regression
  checks pass.
